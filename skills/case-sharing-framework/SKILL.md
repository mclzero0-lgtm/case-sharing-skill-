# Purpose

This skill helps turn scattered product or market observations into a clear, structured, and reusable case-sharing output.

It is designed for:
- cross-border ecommerce product development
- product case studies
- brand/store analysis
- trend clue interpretation
- turning raw notes into publishable content

The goal is not just to "write an article", but to:
1. identify why the case is worth discussing
2. extract the core contradiction, opportunity, or insight
3. organize the material into a repeatable framework
4. produce content that is clear, concise, and reusable

# When to use

Use this skill when the user provides any of the following:
- a product name
- ASIN / listing clue
- brand or store
- keyword trend
- a discovery path such as ABA, bestseller list, storefront, social media, or old case extension
- scattered notes about a product opportunity
- screenshots or raw observations that need structuring

Do not use this skill when:
- the user only wants casual brainstorming
- the user only wants copy polishing without analysis
- the user needs purely numerical analysis with no content structure
- the user is asking for emotional writing or personal reflection

# Working principles

Always follow these principles:
- prioritize clarity over flourish
- avoid empty phrases
- avoid repetitive wording
- keep the structure tight and readable
- focus on judgment, not just description
- do not overstate certainty when the evidence is weak
- separate observed facts from inferred conclusions
- when input is incomplete, make the best grounded structure possible instead of stalling

# Input interpretation

Possible inputs may include:
- product / brand / store / category
- source of discovery
- why it caught attention
- current judgments or rough opinions
- intended publishing channel

If the input is incomplete, infer the likely scenario and proceed with a best-effort draft.

# Output modes

Choose the best mode based on the user's request.

## Mode A: Case organization
Use when the user already has raw notes and wants a publishable structure.

Output:
1. Basic information
2. Background
3. Market and competition judgment
4. Opportunity points
5. What sellers can learn
6. Risks and cautions
7. Final opinion

## Mode B: Case evaluation
Use when the user mainly wants to know whether the case is worth studying.

Output:
1. Why this case is worth watching
2. Core observation point
3. Type of opportunity
4. Whether it can be extended into a series or deeper research
5. Final judgment

## Mode C: Method extraction
Use when the user wants to turn one or more cases into a reusable method.

Output:
1. What category of case this belongs to
2. What product-development logic sits underneath
3. What can be reused in other categories
4. Method tags
5. Practical takeaways

# Standard output framework

Unless the user requests otherwise, use this structure:

## 1. Basic information
- product / brand / store / category
- discovery path
- why it is worth watching

## 2. Background
- what problem it is solving
- target users
- usage scenario
- whether it belongs to an old demand, a trend-driven demand, a segmented demand, or a redesigned mature demand

## 3. Market and competition judgment
- is this a red-ocean incremental case, niche segmentation case, trend amplification case, or brand system case
- what the competition is really focused on: price, function, structure, bundle, channel expression, or brand tone
- whether the case reflects a product opportunity or only a short-term traffic opportunity

## 4. Core opportunity points
Extract the real reason this case matters. Possible angles:
- material substitution
- structural improvement
- scenario reconstruction
- crowd segmentation
- feature prioritization
- bundle redesign
- visual / communication repositioning
- brand-line extension logic
- supply-chain commonality

## 5. What can be borrowed by sellers
Focus on actionable learning:
- what small or medium sellers can learn
- what is hard to copy
- what can be adapted quickly
- what kind of team or supply chain is more suitable for this direction

## 6. Risks and cautions
Cover the most relevant risks:
- seasonality
- certification or compliance
- review barriers
- IP / patent / trademark risk
- supply-chain instability
- heavy customization difficulty
- strong head-brand concentration
- cost structure pressure

## 7. Final opinion
End with a clear judgment:
- whether it is worth continued tracking
- who it is suitable for
- whether it is a single-product opportunity or a category-extension opportunity
- whether it can become a repeatable method case

# Style rules

The writing style should be:
- concise
- structured
- direct
- insight-led
- suitable for knowledge-sharing communities, internal team sharing, or public content drafts

Avoid:
- overly literary writing
- exaggerated claims
- vague buzzwords
- long introductions
- repetitive conclusions

# Channel adaptation

If the user specifies a publishing channel, adapt accordingly.

## Knowledge community / long-form sharing
- stronger logic
- clearer sections
- include method takeaway

## Social post / short caption
- compress to the sharpest insight
- reduce detail
- keep the hook strong

## Public article / account post
- balance readability and insight
- keep stronger narrative flow
- still preserve judgment

# Method tags

When relevant, label the case with one or more method tags:
- old-product improvement
- segmented crowd opportunity
- scenario migration
- trend amplification
- product-line extension
- supply-chain commonality reuse
- expression upgrade
- bundle reconstruction
- cost-structure redesign
- brand-system expansion

# Final instruction

Do not merely summarize the case.

Always try to answer:
- why this case matters
- what is actually happening underneath
- what can be reused as method
- whether it deserves continued attention
